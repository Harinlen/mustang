# 可读代码指南

本文整理自
[`如何编写可读性高的代码`](https://heyitao.com/post/reading-notes-bxkddmdys)，
并补充
[`6 个编写优质干净代码的技巧`](https://gist.github.com/baymaxium/e2a335c5f232023ed01f71416cb0e159)
中适合作为工程检查项的内容。

这不是原文复制，而是把文章里的观点整理成一份可以单独使用的
代码可读性手册。核心标准只有一个：让后来的维护者用尽可能短的
时间理解代码，并且能安全地修改它。

## 为什么可读性重要

代码不是只写给机器执行的，也是写给人维护的。软件的成本不只在
第一次开发，更大量地发生在后续阅读、理解、修改、测试、调试和
交接中。可读性差的代码会把每一次维护都变成考古。

可读代码带来的收益：

- 更容易重新进入一个很久没碰的项目。
- 新成员更容易理解系统，减少 onboarding 成本。
- 团队更容易延续统一的编码模式。
- 修改时更少依赖猜测，降低引入回归的概率。
- 代码本身更接近真实架构，设计意图不只藏在作者脑子里。

## 1. 把信息装进名字里

名字是最小的注释。好的名字能告诉读者：这个值是什么、怎么用、
有什么约束、属于哪个领域。

命名原则：

- 使用更精确的词。`fetch`、`load`、`parse`、`render`、
  `persist`、`broadcast` 往往比 `get`、`handle`、`process`
  更具体。
- 避免空泛名字，例如 `tmp`、`data`、`result`、`info`、
  `manager`。除非作用域极小，而且含义一眼可见。
- 用具体名字代替抽象名字。代码处理的是什么，就尽量命名成什么。
- 把重要细节放进名字：单位、格式、边界、状态、所有权。
  例如 `timeout_seconds`、`max_retries`、`raw_payload`、
  `parsed_message`。
- 作用域越大，名字越要完整。局部循环里的 `i` 可以接受；跨模块
  API 的名字必须能独立说明语义。
- 少用项目内部缩写。除非它已经是代码库中稳定、明确的约定。
- 用命名格式传递一致含义，例如前缀、后缀、大小写、下划线的用法
  在同一项目中应保持一致。

检查问题：读者只看名字，能不能知道这个值最重要的信息？

## 2. 避免会被误解的名字

一个名字不仅要准确，还要避免常见误读。

范围命名：

- `min` / `max` 表示下限和上限。
- `first` / `last` 更适合包含端点的范围。
- `begin` / `end` 常用于可能右开区间的范围。
- `limit` 容易有歧义，只有边界语义非常清楚时再用。

布尔命名：

- 布尔值应该像判断句：`is_enabled`、`has_token`、
  `should_prompt`、`can_execute`、`needs_refresh`。
- 尽量避免否定式布尔名，例如 `disable_ssl`。多数时候
  `ssl_enabled` 或 `use_ssl` 更容易读。
- 避免条件里出现双重否定。

读者预期：

- `get_*`、`size` 这类名字通常让人以为操作很轻、无副作用。
  如果函数会 I/O、阻塞、修改状态，应该换更诚实的动词。
- `filter` 可能表示保留匹配项，也可能表示过滤掉匹配项。必要时
  用 `select_*`、`exclude_*`、`remove_*` 消除歧义。

检查问题：读者会不会根据这个名字做出合理但错误的假设？

## 3. 保持整洁且一致的格式

格式的目的不是炫技，而是让结构一眼可见。具体风格不如团队一致
重要。

格式原则：

- 常规格式交给 formatter。
- 用空行把较长函数分成逻辑段落。
- 对重复项选择有意义的顺序，并持续使用：字母顺序、生命周期顺序、
  依赖顺序、请求到响应的顺序、优先级顺序等。
- 只有在能明显提升浏览效率时才手动对齐，不要和 formatter 对抗。
- 密集字面量、长参数列表、嵌套调用应适当换行，让层次清楚。

检查问题：读者在读每个 token 前，能不能先看出代码形状？

## 4. 写能传递意图的注释

注释应该帮助读者知道作者当时知道的事，而不是重复代码已经说清楚
的事。好名字优先于好注释。

不要写的注释：

- 重复代码表面行为的注释。
- 用注释解释糟糕命名；应先改名。
- 注释掉的旧代码；版本控制已经保存历史。
- 大段没有维护价值的背景故事。

应该写的注释：

- 为什么选择这个方案，而不是另一个看似更直接的方案。
- 协议、供应商、文件格式、并发、性能、安全、兼容性等外部约束。
- 不符合直觉的边界条件。
- 常量为什么取这个值。
- 临时折中，以及之后如何判断是否还能保留。
- 文件级或类级的全局说明：主要部分如何协作。
- 难读代码块前的一句意图总结，防止读者迷失在细节里。

写注释时要精确紧凑：

- 少用 `this`、`it` 这种指代不清的词。
- 描述高层意图，不逐行翻译代码。
- 行为复杂时，用小输入/输出例子解释。
- 用具体词代替 `fix`、`stuff`、`special` 这类泛词。

检查问题：这条注释有没有说明代码本身说不出来的信息？

## 5. 让控制流容易读

好的控制流是线性的、自然的。读者不应该在脑子里维护很多层嵌套
才能知道下一步发生什么。

控制流原则：

- 用 guard clause 和提前返回减少深层嵌套。
- 优先呈现正确、简单、常见的路径。
- 把异常、无效、少见分支收束在清楚的位置。
- 避免复杂三目表达式；小的 `if` 往往更清楚。
- 少用项目中不常见、会打断阅读习惯的控制结构。

比较表达式：

- 通常把变化的值放左边，稳定的值放右边。
- 边界判断要让包含/排除关系足够明显。

检查问题：读者能否不模拟所有分支，就追踪出成功路径？

## 6. 拆分超长表达式

大多数人同时只能稳定处理少量信息。一个表达式如果塞进太多概念，
读者就需要反复回看。

拆分方法：

- 为有业务含义的中间结果引入解释变量。
- 把复杂布尔条件拆成命名良好的谓词。
- 使用德摩根定理时，只在它让条件更容易读时才用。
- 重复子表达式可以抽成 helper，但 helper 必须有清晰名字和契约。

也要避免无意义变量：

- 如果临时变量只是复述一个显而易见的表达式，它会让代码更长。
- 如果变量只用一次且名字模糊，要么内联，要么换成真正解释意图的
  名字。

检查问题：这个表达式是否要求读者同时记住超过三四个概念？

## 7. 用变量降低理解负担

变量的价值在于命名概念、缩小范围、降低状态理解成本。

变量原则：

- 减少不能改善可读性的临时变量。
- 变量尽量靠近第一次使用的位置。
- 作用域越小越好。
- 能单次赋值就不要引入可变状态。
- 避免全局或长生命周期可变状态，除非所有权和生命周期非常清楚。
- 如果一个值会变化，变化阶段应当明显。

检查问题：这个变量在被使用前，有多少地方可能改变它？

## 8. 抽取不相关的子问题

难读函数常常把多种工作混在一起：通用解析、领域策略、I/O、校验、
格式化、状态修改、错误处理。拆分的目的不是让文件更多，而是让每个
部分有清楚的问题边界。

适合抽取的情况：

- 通用逻辑和项目专有策略混在一起。
- 解析、校验、转换、格式化有稳定契约。
- 调用方应该关注操作主线，而不是底层机械细节。
- 一个代码块解决的问题和周围代码不是同一类。

不适合抽取的情况：

- 只是把代码移动到别处，没有命名出新概念。
- helper 的名字比原代码还难懂。
- 抽取后调用方必须跳来跳去才能读懂主线。

检查问题：这段代码解决的问题，和它周围代码解决的是同一个问题吗？

## 9. 一次只做一件事

如果一段代码难读，先用自然语言列出它正在做的任务。列表里如果有
多个不相关任务，就应该拆成函数、类，或至少拆成清楚的逻辑段落。

函数做太多事的信号：

- 很难不用 `and` 给它命名。
- 需要长注释解释调用方式。
- 同时做校验、转换、持久化、发事件。
- 局部变量很多，而且生命周期互不相关。
- 错误路径来自不同层次。

检查问题：这个函数能不能用一个清楚的动词加一个对象概括？

## 10. 先用自然语言解释代码

当实现变得纠结时，先把行为用自然语言说清楚。橡皮鸭调试的价值在于：
口头解释会暴露隐藏分支、混杂职责、缺失命名和偶然复杂度。

快速流程：

1. 说清楚代码应该做什么。
2. 列出输入和输出。
3. 列出分支和失败情况。
4. 给每个子任务命名。
5. 只有在能让代码更清楚时，才把这些名字变成函数或逻辑块。

检查问题：现在的代码是否读起来像刚才那段解释？

## 11. 少写代码

代码越多，需要开发、测试、文档、调试、安全审查和维护的东西就越多。
最干净的代码常常是不存在的代码。

减少代码的方法：

- 删除不必要功能和死路径。
- 避免为猜测中的未来需求做抽象。
- 优先解决当前版本的真实问题。
- 熟悉并复用标准库和项目已有 API。
- 当共享模式已经明显时，再删除重复 helper，抽出共同实现。

检查问题：删掉这段代码会不会让系统更容易理解，同时不丢失必要行为？

## 12. 测试也要可读

测试是系统行为的文档。好的测试不仅能失败，还能让人快速知道坏在哪。

可读测试原则：

- 测试名说明被验证的行为。
- 输入尽量简单，但要完整覆盖目标路径。
- 断言可观察行为，而不是偶然实现细节。
- 失败信息应指向具体破坏的行为。
- 测试应便于修改和新增。
- 不要为了 100% 覆盖率牺牲测试价值；覆盖率是信号，不是目标本身。

检查问题：半年后这个测试失败时，维护者能不能从测试名和失败信息知道
应该看哪里？

## 13. 保持风格一致，并定期检查

干净代码不是一次性成果，而是持续维护的状态。团队代码库越统一，
新代码越容易自然延续好模式。

维护原则：

- 默认遵循当前项目约定。
- 新风格先在小范围试验，确认更好后再推广。
- 如果新模式确实更清楚，要文档化，并一致应用。
- 修改旧代码时做相关的小幅可读性改善，但避免无关的大规模 churn。
- 定期回看常用模块，避免代码在日常迭代中慢慢腐化。

检查问题：这次改动让下一个贡献者更容易还是更难遵循本地模式？

## 快速检查清单

提交或结束任务前，按这些问题扫一遍：

- 读者能否快速理解这段代码的目的？
- 名字是否包含关键事实，并避免误导？
- 格式是否符合项目习惯，结构是否一眼可见？
- 注释是否解释意图、约束或意外行为？
- 控制流是否尽量线性、嵌套是否足够浅？
- 复杂表达式是否拆成了有名字的概念？
- 变量作用域是否足够小，生命周期是否清楚？
- 通用逻辑和项目策略是否分开？
- 每个函数是否只做一件主要的事？
- 是否能先用自然语言解释，再让代码贴近这个解释？
- 有没有可以删除的代码、功能或抽象？
- 测试是否像行为文档一样清楚？
- 代码是否延续了当前模块的本地风格？
